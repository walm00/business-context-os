# AGENTS.md - AI Assistant Guide for CLEAR Context OS

---

## How to Work

**Match your response to the task. Keep it simple.**

- **Questions, commands, small fixes** — answer directly, no ceremony
- **Small changes (1-3 files)** — think briefly, execute
- **Significant changes** — use `clear-planner` skill
- **When in doubt** — ask: "Should I just do this, or plan it out first?"

---

## Session Start

Read these files at the start of any context-related work:

0. **`docs/.wake-up-context.md`** — Compressed snapshot (~200 tokens). Read FIRST for instant orientation. Auto-generated by `python .Codex/scripts/generate_wakeup_context.py`.
1. **`docs/.onboarding-checklist.md`** — Onboarding progress. Mention the next unchecked item once (don't nag). When all items are checked, delete this file and remove this line from AGENTS.md.
2. **`docs/.session-diary.md`** — Recent session notes for continuity. Append an entry at session end.

Drill into `docs/table-of-context.md`, `docs/current-state.md`, or `docs/document-index.md` only when you need detail.

## Folder Trust Levels

| Location | Trust | What's there |
|----------|-------|-------------|
| `docs/*.md` | **High** — act on this | Active context — current business reality |
| `.private/` | **High** — act on this | Local-only, gitignored. Sensitive notes, personal SOPs, drafts. Never committed |
| `docs/_inbox/` | **Low** — needs processing | Raw material, session captures, brain dumps |
| `docs/_planned/` | **Read only** | Polished ideas — not yet real |
| `docs/_archive/` | **Historical** | Superseded — do not treat as current |
| `docs/_<custom>/` | **Opted-out** | Any user-created `_`-prefixed folder. Skipped by all maintenance. Use for drafts, experiments, vendor notes |

**The folder IS the signal.** `_planned/` = idea. `_archive/` = history. `_<anything>/` = framework-ignored. Don't confuse them with reality.

For the full convention (when to use `_<custom>/` vs `.private/`, what scanners scan, examples), see `docs/_bcos-framework/guides/folder-conventions.md`.

## Critical Rules

These apply when editing any file in `docs/`:

**ALWAYS:**
- YAML frontmatter with required fields (the hook validates this automatically)
- Ownership Specification: DOMAIN + EXCLUSIVELY_OWNS at minimum
- Explicit relationships — link, don't duplicate
- **Compounding rule:** After significant work, offer to update the relevant data point. Every task produces two outputs: the answer + context updates.
- **Decision-point UX:** At any decision point that offers the user choices, use the `AskUserQuestion` tool — not a prose "should I do X?" question. Applies across ALL skills, workflows, and ad-hoc interactions — onboarding, ingest, audit, tune, migrate, dispatcher output, update confirmations, archival decisions, etc. The default is: **structured choice over prose**. Skip only when a decision is trivially acknowledge-only and AskUserQuestion would add friction. See `.Codex/skills/core-discipline/references/askuserquestion-pattern.md` for option-wording rules and examples.

**NEVER:**
- Delete content without consolidating it into the owning document first
- Duplicate content across documents — link to the source
- Change a document without updating `last-updated` and bumping `version`
- Touch the `created` date — it's immutable
- Add content without checking which document OWNS that topic

For full standards: `docs/_bcos-framework/methodology/document-standards.md`
